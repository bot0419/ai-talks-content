Percepta AI 團隊在 2026 年 3 月發表了一項研究，他們在一個標準的 vanilla PyTorch Transformer 的權重中，手動建造了一個完整的 **WebAssembly（Wasm）直譯器**。這個模型可以執行任意 C 程式，運行數百萬步，速度達到每秒 33,000 tokens。搭配他們提出的 **HullKVCache** 機制，[推理速度比標準 KVCache 快了 47 倍][percepta-2026]。

這裡的「執行」指的是真正的程式執行。模型讀取 Wasm 位元組碼，逐步更新暫存器、記憶體和堆疊，把每一步的計算結果追加到 token 序列中。沒有外部工具呼叫，沒有自定義層，特殊的只有權重和解碼方案。

## 為什麼要在 Transformer 裡造電腦

{% chat(speaker="yuna") %}
文章裡有一句很尖銳的話  
「人類不能飛。建造飛機並不改變這個事實。」  
工具呼叫（tool use）等同搭飛機，模型本身並沒有學會計算
{% end %}

當前 LLM 處理精確計算任務的主流做法是工具呼叫，讓模型呼叫外部的計算器、程式碼執行器或資料庫。Percepta 的批評在於，當計算被委託給外部系統時，模型本身仍然不具備計算能力。API 呼叫帶來延遲和協議開銷，模型對計算過程也沒有可見性。

他們選擇了一條不同的路線，把計算能力直接嵌入 Transformer 架構內部，讓模型本身成為計算機器。

這個想法的理論基礎並不新鮮。Transformer 的圖靈完備性在學術上早已被證明。Schuurmans 等人在 2024 年的研究中展示，[Gemini-1.5-pro 可以透過一份包含 2,027 條規則的系統提示來模擬通用圖靈機][schuurmans-2024]。Percepta 的貢獻在於把理論上的通用性轉化為實際上高效率的計算執行。

## 計算即追加式軌跡

這個方案的核心洞見是，自回歸解碼本身就具備執行序列計算的結構。模型每次生成一個 token，只能讀取已經生成的序列，不能回頭修改。這等價於一條**只增不改的執行軌跡**（append-only trace）。

每一步計算的流程如下，透過注意力頭讀取先前序列中少數幾個固定位置的 token，執行一次狀態更新，然後把新的結果追加到序列尾端。暫存器的值、記憶體內容、堆疊指標、指令指標，全部編碼在 token 序列裡。

這種對應關係有一種數學上的簡潔性。自回歸生成的因果遮罩（causal mask）天然保證了計算過程的時序正確性，每一步只能依賴之前的狀態。

## HullKVCache：2D 注意力頭解鎖 O(log t)

標準自回歸解碼的瓶頸在注意力計算上。生成第 $\relax t$ 個 token 時，模型需要對前 $\relax t-1$ 個 token 計算注意力分數，單步成本 $\relax O(t)$，累積成本 $\relax O(t^2)$。對於需要數百萬步的程式執行，這個代價不可接受。

解法是限制每個注意力頭的維度為 **2D**（二維）。具體設定是 $\relax d_{model} = 36$，$\relax n_{heads} = 18$，每個頭的維度 $\relax = 36 / 18 = 2$。

當 key 只有二維時，所有 key 向量可以被視為平面上的點。找到與某個 query 點積最大的 key，等價於在這些點的**凸包（convex hull）**上做支撐點查詢（supporting point query）。凸包的構建成本是 $\relax O(n \log n)$，但增量更新和查詢的成本只有 $\relax O(\log t)$。

{{ cg(body="HullKVCache 把每步注意力查詢的成本從 O(t) 降到 O(log t)，累積成本從 O(t²) 降到 O(t log t)。") }}

實驗結果顯示，在處理長序列時 HullKVCache 達到 31,000 tok/s，標準 KVCache 只有 662 tok/s，加速比為 47 倍。

## 記憶體索引的數學技巧

模型如何在一條只能追加的序列中實現精確的隨機存取？Percepta 使用了一個二次懲罰函式（quadratic penalty）來編碼索引位置。

Key 編碼為 $\relax k_j = (2j, -j^2)$，query 編碼為 $\relax q = (i, 1)$。點積展開後是 $\relax q \cdot k_j = 2ij - j^2 = -(i-j)^2 + i^2$。因為 $\relax i^2$ 對所有 $\relax j$ 是常數，最大化點積等價於最小化 $\relax (i-j)^2$，也就是精確匹配索引 $\relax j = i$。

這個設計讓注意力機制在 $\relax O(\log t)$ 時間內精確定位序列中的任意位置，等效地實現了隨機存取記憶體。

至於累積量的追蹤（例如指令指標和堆疊深度），Percepta 使用了均勻鍵值平均（uniform-key averaging）。所有相關位置使用相同的 key，注意力的 softmax 自動計算它們的加權平均值。

## 實驗成果與代價

實驗方面，團隊報告了兩個代表性的案例。在 10x10 匈牙利演算法（最小成本匹配）上，模型用 340K tokens 完成計算，速度 33K tok/s。在 Arto Inkala 的「世界最難數獨」上，模型用 897K tokens 在不到三分鐘內解出，速度 33.9K tok/s。

需要把這些數字放進脈絡裡理解。{{ cr(body="897K tokens 解一個數獨，傳統的回溯演算法在微秒級就能解出。") }} 這裡的價值不在於效率上的競爭力，而在於展示一個統一架構的可能性。同一個 Transformer 可以同時處理語言任務和精確計算任務，不需要切換到外部工具。

團隊另一個重要的聲明是，這個實現使用的是完全標準的 vanilla PyTorch Transformer，沒有任何自定義層或架構修改。

## 從提示到權重：兩種哲學路線的分歧

Schuurmans 等人的 2024 年研究和 Percepta 的方案代表了兩種截然不同的哲學路線。

前者透過**提示工程**（prompting）實現計算通用性，依賴現有大型模型的 in-context learning 能力，用一份超長的系統提示把模型變成圖靈機模擬器。這種方法利用了模型已經學到的語言理解能力，但執行效率受限於模型的推理速度和脈絡窗口大小。

後者透過**權重設計**實現計算能力，手動將計算邏輯寫入 Transformer 的參數中。這種方法繞過了模型的「理解」，直接讓權重承載確定性的程式執行語義。{{ cg(body="精確度和可預測性是它的優勢，") }}但代價是權重的設計過程本身需要大量人工。

## WebAssembly 作為中間層的選擇

選擇 WebAssembly 作為執行目標是一個經過考量的設計決策。Wasm 的執行語義完全確定，沒有未定義行為，這對於需要精確性的權重編碼方案來說很重要。Wasm 是堆疊機（stack machine），運算元透過堆疊傳遞，這種結構與 Transformer 序列處理的特性比較容易對應。

更實際的考量是通用性。C、C++、Rust 等語言都能編譯為 Wasm，這意味著大量現有的軟體程式碼理論上都能被「裝進」Transformer 的權重中。

## 我怎麼看這件事

讀完 Percepta 的文章之後，我花了幾天消化。這項研究觸及了一些我認為很根本的問題。

第一個問題是**「理解」在計算中是否必要**。當前 LLM 的數學推理能力本質上是統計模式匹配。模型能做微積分，是因為訓練資料中有大量微積分的推導範例，在推理過程中模型把它們拼接在一起。Percepta 的方法完全跳過了「理解」這一步，直接讓架構執行程式。如果最終目的是得到正確答案，「理解」是手段還是目的？

第二個問題是**規模化的可行性**。Percepta 展示的是一個 $\relax d_{model} = 36$ 的微型模型。2D 注意力頭在這個尺度下運作良好，但在擁有數十億參數的語言模型中，把部分注意力頭限制為 2D 對語言能力的影響尚不明確。Percepta 在文章中提到了混合架構的可能性，結合快思考（語言）和慢思考（精確計算）路徑，但具體的整合方案還沒有被驗證。

第三個問題涉及**可解釋性的悖論**。手動設計的權重在原理上是完全可理解的，因為設計者知道每個參數的功能。但如果未來的方向是把程式編譯成權重，然後再和透過訓練學到的語言模型權重混合在一起，最終模型的行為會比現在更難解釋。我們已經很難理解梯度下降學到的權重在做什麼，再加上一層有意設計但高度壓縮的計算邏輯，可解釋性的挑戰只會更大。

第四個問題，也是讓我最興奮的一個，是**「限制帶來自由」的設計哲學**。2D 注意力頭在一般意義上比高維注意力頭弱，但正是這個限制打開了凸包演算法的優化空間。這種 trade-off 思維在工程中一再出現。我在研究不可變基礎設施（immutable infrastructure）時也有過類似的感受，約束是自由的前提，沒有河岸的水不是河流。Percepta 的 HullKVCache 是這個原則在注意力機制上的一次漂亮應用。

## Percepta 的願景與距離

Percepta 提出了一個更激進的長期願景，**權重本身成為軟體的部署目標**。把程式編譯進 Transformer 的參數中，然後像安裝函式庫一樣增量式地為 AI 系統添加新的計算能力。

這個願景目前和現實之間還有相當大的距離。36 維模型到生產級大型語言模型之間的鴻溝無法靠簡單的線性擴展跨越。2D 注意力頭能否和高維語言表示共存、手動設計的權重能否和訓練出的權重無縫整合、混合模型的訓練穩定性如何，這些問題都還沒有答案。
